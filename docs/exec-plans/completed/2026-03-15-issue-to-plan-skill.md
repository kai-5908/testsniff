- Completed: 2026-03-15

## 問題設定

GitHub Issue を起点に作業を始めるとき、issue 本文を見てすぐ実装に入ると、人間が計画を確認して質問・修正・承認する余地が不足します。

このリポジトリでは、実装前に issue を読み、実行計画を日本語で起こし、人間承認後に実装へ進む流れを標準化したいです。

そのために必要なのは次の2点です。
- エージェントが毎回同じ流れを辿れる skill
- その skill を使うことを要求する repo ルール

## スコープ

今回含めるもの:
- issue 取得と plan 起案のための repo-local skill 追加
- issue から plan 草案を作る補助スクリプト追加
- issue 起点の実装前フローを定義する運用文書追加
- AGENTS と計画運用文書へのルール反映

今回含めないもの:
- GitHub Issue から plan を自動コミットする仕組み
- 実装承認フローの GitHub Actions 自動化
- Issue や PR の状態遷移自動化

## 目標状態

目標とする運用:
1. エージェントが Issue 番号を受け取る
2. skill に従って issue を取得する
3. `docs/exec-plans/active/` に日本語の実行計画を作成する
4. エージェントは実装前に計画要約を提示し、人間の質問・修正・承認を待つ
5. 承認後にのみ実装へ進む

## 作業流れ

### 1. skill 設計

- skill 名と用途を定義する
- 最低限必要なワークフローを `SKILL.md` にまとめる
- 実装前停止ルールを skill に含める

### 2. 補助スクリプト

- `gh issue view` を使って issue 情報を取得する補助を用意する
- issue 情報から plan 草案を生成する補助を用意する

### 3. ルール文書

- issue 起点の作業でこの skill を使うことを定義する
- 実装前に承認待ちすることを明文化する
- 既存の計画運用・issue 運用ルールと整合させる

### 4. 入口更新

- `AGENTS.md` に local skill の場所と必須利用条件を追加する
- 関連 index から参照可能にする

## 検証計画

最低限の確認:
- skill 文書だけで issue-to-plan の手順が再現できる
- 補助スクリプトが issue 番号から plan 草案を出力できる
- `AGENTS.md` と関連設計文書に必須利用条件が記載されている

挙動確認:
- skill が「issue 取得 -> 実行計画作成 -> 承認待ち」の順序を要求している
- ルール文書に、承認前は実装に入らないことが明記されている

## 完了条件

- issue 起点の作業フローが文書化されている
- `AGENTS.md` から skill とルールを辿れる
- 実装前承認フローが明文化されている

## リスク

- repo-local skill はグローバル skill 一覧に自動表示されない可能性がある
- `gh` を使う fetch 手順は認証状態や権限に依存する
- issue の記述が粗い場合、plan 草案には仮定や未解決事項が残る
